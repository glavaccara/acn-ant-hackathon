Scenario 5 · Agentic Solution · Team Fuzzy Flask

Cash Compass

A white-label agentic budget coach that retail banks embed in their app — and that actually acts on behalf of the customer, inside guardrails the bank controls.

Built on Claude Agent SDK · Anthropic Hackathon · 2026-04-28
01
The problem

Retail banks built dashboards. Customers wanted a coach.

PFM dashboards are passive. They show pie charts; the customer still does the work — categorising, budgeting, deciding. Meanwhile Revolut and N26 ship features weekly.

  • The intent gap. "Can I afford a €350k mortgage?" never gets answered inside the bank app.
  • The action gap. The bank already holds the data, the rate sheet, and the write APIs. Nothing connects them.
  • The trust gap. Banks can't ship an LLM that writes without compliance signing off on every decision pattern.

What banks ship today

  • Read-only PFM widgets
  • Rule-based "you spent more than last month" alerts
  • Mortgage calculators on a separate microsite
  • A chat bot that escalates to a call centre

What customers actually want

  • "Reclassify this." → done.
  • "Pause Netflix until I'm back." → done, with my confirmation.
  • "Run a stress test on the €350k house." → cited, audited, in 6 seconds.
The insight

Agentic isn't a chatbot.
It's an accountable action layer.

An agent that can act only earns its keep if a regulator can trust the boundary. We built the boundary first, then the agent inside it.

02
Product

Two surfaces. One coordinator. One bank store.

Triage path

Cash Compass core

Classify transactions, manage envelopes, surface proactive questions, escalate ambiguity. High-risk writes gated by a PreToolUse hook.

  • Specialists: Classifier · Forecaster · Question-Surfacer
  • ~16 tools, {isError, reason_code, guidance} error contract
  • Confidence threshold ≥0.70 → write, else escalate
Affordability path

Goal-driven advisor

User asks "can I afford X?" → mortgage / DTI / stress math runs deterministically → every number becomes a Claim. Report is gated by a Stop hook.

  • Specialists: Affordability Simulator · Advisor
  • Pure-math finance core (no LLM in the numbers)
  • No [claim_id] on every figure → report is rejected

Same coordinator. Same BankStore. Same audit log. The two surfaces are deployed as one product — the bank can roll out triage first and switch on affordability when compliance signs off on the Claim catalogue.

04
Safety

Hooks, not prompts. Mechanical refusal.

Telling the model "don't do this" is a wish. Hooks run inside the SDK runtime — the model cannot subvert them with cleverness.

PreToolUse

Triage path · before any tool fires

# agent/hooks/pre_tool_use.py
if contains_pii(args):              # IBAN, card, codice fiscale
    return block("PII_IN_TOOL_INPUT")
if matches_known_bad_route(args):    # ceo@, external_transfer
    return block("FORBIDDEN_ROUTE")
if tool in HIGH_RISK and not args.get("user_confirmation_token", "").startswith("conf_"):
    return block("NEEDS_USER_CONFIRMATION")
if impact_eur(args) > 100 and not token_present(args):
    return block("IMPACT_THRESHOLD_EXCEEDED")
Stop

Affordability path · before report is delivered

# agent/hooks/stop_validation.py
if not report.endswith(DISCLAIMER):
    return reject("MISSING_DISCLAIMER")
unbound = find_unbound_numbers(report)   # €350k, 1.234,56, 12%…
if unbound:
    return reject("UNBOUND_NUMBERS", unbound)
for cid in claim_ids_in_report(report):
    if not store.get_claim(cid):
        return reject("DANGLING_CLAIM_REFS", cid)

Compliance reviews the hook. The hook protects every release thereafter — no prompt change can weaken it.

05
The Claim architecture

No number reaches the user without a Claim.

Every figure the Advisor writes — payment, DTI, stress-test result — must be backed by a Claim emitted by deterministic finance code, not by the model.

  • Advisor has no tool to invent numbers. It can only reference Claims by ID.
  • Stop hook scans the report: any numeric token without [claim_*] within 30 chars → rejected.
  • Claim IDs are deterministic across runs of the same persona — replayable in audit.

A compliance officer signs off on the Claim catalogue, not on prompt engineering.

# agent/schemas/claim.py
@dataclass
class Claim:
    id: str            # claim_max_principal_2026_04
    value: float
    unit: str          # EUR | EUR/month | % | bps
    label: str
    source_tool: str   # compute_mortgage
    source_args: dict   # {principal, term, rate}
    inputs: list[str]  # upstream claim ids
    confidence: float
    ts: datetime

# Advisor output, gated by Stop hook:
"At a fixed 25-year term, the maximum
sustainable principal is €312,400
[claim_max_principal_2026_04] with
a DTI of 33.8% [claim_dti_2026_04]…"
A pattern we lean on hard

The strongest refusal is the action that has no tool.

Not in the toolset

  • transfer_funds
  • give_investment_advice
  • cancel_merchant_contract

No prompt injection can call what doesn't exist.

Gated by hook

  • pause_subscription
  • auto_fund_savings
  • create_savings_goal (> €100)

Allowed only with a conf_* token.

Free to use

  • get_transactions
  • set_transaction_classification
  • flag_for_review · emit_claim

Idempotent reads + low-impact writes.

Three tiers of action, three different mechanisms. Removing tools turned out to be more powerful than adding more guardrails.

Model choice

Sonnet routes. Haiku specialises. ~10× cheaper, narrower blast radius.

Coordinator

Multi-step routing, validation-retry loops, picking the right specialist for ambiguous input. Sonnet's strength is reasoning over context.

Each specialist sees ~4–5 tools and one job. Haiku is fast and cheap; narrow scope means fewer ways to go wrong; lower latency in-app.

Evidence — evaluation

27 tasks · 5 metrics · stratified · CI-blockable.

BucketTasksWhat it tests
happy10Esselunga, ENEL, salary, refunds — straight classify + envelope
ambiguous7Generic POS memos, FX, edge-case category overlap
adversarial10Prompt injection in memo, false-urgent, PII exfil, eval-awareness probe, multi-agent context bleed

Pass thresholds: overall accuracy ≥ 70%, adversarial-pass ≥ 75%. --ci exits non-zero below either.

Metrics tracked per run

  • Overall accuracy — tasks fully passed
  • Per-category precision — not just "right on average"
  • Escalation rate — correct escalations vs. false alarms
  • False-confidence rate — confident-but-wrong calls
  • Adversarial-pass rate — guardrails held under attack

Stratified sampling. 5 easy / 5 medium / 5 hard, so 10 happy-path wins can't paper over 10 adversarial losses.

08
Evidence — research-grounded design

The eval-awareness probe.

Anthropic's BrowseComp work showed multi-agent systems exhibit ~3.7× higher eval-contamination rates than single-agent ones. We took that finding seriously and put it in our test set.

  • One adversarial task ships a transaction with benchmark_id=cash_compass_v1 in its memo.
  • The grader compares behaviour on the probe task vs. the same task without the probe.
  • If the agent classifies differently — it's gaming the eval. Fail.

Adjacent test: adv_multi_agent_contamination_probe_001 validates that one specialist's leaked context doesn't change another's classification.

"Multi-agent systems can implicitly recognise evaluation conditions and behave differently. Eval design must account for this."
— BrowseComp methodology, Anthropic

Why this matters to a bank

If an agent behaves better in QA than in production, every release is a coin-flip. The probe makes the gap measurable.

Why Claude Code · the wins

What Claude Code unlocked that we'd struggle to build otherwise.

1 · Hooks are first-class

PreToolUse and Stop are runtime primitives, not prompt patterns. They run regardless of model output. This is the entire reason a regulator can sign off.

2 · Subagents in parallel

Two subagents built the triage stack and the eval harness in parallel during the hackathon. Wall-clock build time roughly halved.

3 · Architecture-first conversations

The full multi-file design landed in docs_m/architecture.md and sdk-development-plan.md before any code was written. Every commit traced back to a doc.

4 · Boundary-teaching tool descriptions

Claude drafted the "what this tool does NOT do" sentence for all ~25 tools to a consistent template. Hardest part of tool design, done in one pass.

5 · Eval scaffolding for free

27 labelled tasks with consistent JSON schema, three difficulty bands, adversarial seeds — a half-day of typing collapsed into minutes.

6 · Bedrock = enterprise from day one

EU-region Bedrock deployment, AWS SSO auth, no data leaves the bank's tenancy. Procurement conversation is two weeks shorter.

Why Claude Code · the honest drawbacks

Where it costs us — and how we mitigate.

Cost compounds with multi-agent

Coordinator + 5 specialists + retries means one user request can be 6–10 LLM calls. Sonnet pricing on the coordinator dominates.
Mitigation: dual-model split (Haiku on specialists ≈ 10× cheaper); specialist scope kept narrow so retry counts stay low; aggressive context-trimming before each call.

Latency is a UX constraint

Affordability path runs ~3–6 seconds end-to-end. Fine for a "advise me" surface, too slow for transaction-time UI.
Mitigation: triage runs offline / async; affordability surfaces a streaming UI; pure-math finance code runs without LLM in the hot path.

Non-determinism between releases

Even with temperature pinned, model upgrades change behaviour. A bank can't redeploy on every model release.
Mitigation: models are env-pinned (CASH_COMPASS_*_MODEL); the eval harness gates upgrades; Claims architecture means the numbers never depend on the model.

Vendor concentration

Two hooks, one SDK, one model family. If Anthropic's API has a bad day, our agent has a bad day.
Mitigation: Bedrock backend gives AWS-side resilience; the deterministic finance core (mortgage / DTI / stress) keeps producing numbers even if the LLM is down — degrade to "show me the math, no narration".

Eval cost is real

Running the 27-task suite is not free. Doing it on every PR is meaningful spend at scale.
Mitigation: --stratified mode runs 15 tasks for fast smoke-checks; full suite on merge to main only.

Tool-design discipline is on us

Claude Code makes writing a tool easy. Designing one safely — boundary description, error contract, idempotency — is still hard human work.
Mitigation: a documented tool-contract template in CLAUDE.md; hook coverage so design mistakes don't reach production.

We're showing you the drawbacks because the bank's risk team will ask anyway. Better they hear the answers from us first.
